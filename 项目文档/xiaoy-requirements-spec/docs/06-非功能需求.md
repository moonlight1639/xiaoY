# 非功能需求

## 1. 性能需求

### 1.1 前端性能

- 常规页面应在可接受时间内完成首屏渲染
- 普通列表页数据加载应尽量控制在 3 秒内
- 聊天页面切换会话时应避免明显卡顿

### 1.2 后端性能

- 常规查询接口应保持稳定响应
- 分页查询应支持基础性能优化
- 流式聊天接口应尽快返回首段内容，减少等待感

### 1.3 文件处理性能

- 文件上传应支持中等体量文件
- 上传过程失败时应及时返回错误信息

## 2. 安全性需求

### 2.1 身份认证

- 系统应支持基于 `token` 的身份认证
- 受保护接口必须校验认证信息

### 2.2 数据安全

- 用户密码不得以明文形式在页面或日志中暴露
- 敏感配置应通过环境变量或配置文件管理
- 上传接口应校验文件有效性

### 2.3 接口安全

- 应具备基本参数校验能力
- 应防止未登录用户访问受保护接口
- 后续可继续增强越权校验、限流与日志审计

## 3. 可用性需求

- 关键页面应有明确导航
- 操作失败时应有可理解提示
- 页面加载中应有基础反馈
- 404 页面应具备兜底能力

## 4. 可靠性需求

- 后端服务异常时不应导致前端整体崩溃
- 流式聊天异常时应能反馈失败原因
- 中间件不可用时应便于定位故障点

## 5. 可维护性需求

- 前后端代码需分层清晰
- 公共请求逻辑应统一封装
- 页面、服务、类型定义应具备明确命名
- 文档需持续更新，反映真实实现状态

## 6. 可扩展性需求

- 前端页面结构应支持新增模块和路由
- 后端应支持继续增加实体、接口和AI能力
- 当前技术架构应能平滑扩展推荐、知识库、移动端等能力

## 7. 兼容性需求

### 7.1 前端兼容性

- 应兼容主流现代浏览器
- 在常见桌面端分辨率下应正常显示

### 7.2 后端兼容性

- 应支持 Windows 本地开发环境
- 应支持 Linux 服务器部署环境

## 8. 可测试性需求

- 核心模块应便于单元测试和接口测试
- 关键接口应可通过接口工具独立验证
- 文档中应明确主要验收用例和测试范围
