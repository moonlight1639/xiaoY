# 详细设计文档

## 1. 文档目的

本文档在需求分析基础上，对小Y项目的系统结构、模块划分、核心流程、数据设计和关键技术方案进行细化说明，作为开发实现、联调测试和后续维护的依据。

## 2. 系统总体设计

## 2.1 总体架构

小Y项目采用前后端分离架构：

- 前端：`xiaoy-ui`
- 后端：`xiaoY`
- 数据层：MySQL + MongoDB + Redis + MinIO
- AI层：LangChain4j 对接 DeepSeek / 通义千问等模型能力

系统总体逻辑如下：

1. 用户通过前端页面发起操作
2. 前端通过统一 HTTP 模块调用后端接口
3. 后端 Controller 接收请求并分发到 Service
4. Service 调用 Mapper、Repository 或工具类完成处理
5. 结构化数据落库到 MySQL
6. 聊天会话与消息存储在 MongoDB
7. 登录态缓存到 Redis
8. 上传文件保存到 MinIO
9. 后端统一返回 `Result` 响应对象

## 2.2 分层设计

### 前端分层

前端按以下层次组织：

- `pages/`：页面级组件
- `components/`：公共组件
- `router/`：路由配置
- `services/`：接口调用层
- `store/`：状态管理
- `types/`：数据类型定义
- `assets/`：静态资源

### 后端分层

后端按以下层次组织：

- `controller/`：接口入口层
- `service/`：业务接口层
- `service/impl/`：业务实现层
- `mapper/`：MyBatis-Plus / Mapper 层
- `entity/`：实体对象层
- `repository/`：MongoDB 仓储层
- `config/`：配置层
- `interceptor/`：拦截器层
- `assistant/`：AI能力封装层
- `utils/`：通用工具层
- `common/`：统一返回、常量、异常等公共层

## 3. 前端详细设计

## 3.1 路由设计

系统主要路由如下：

| 路由 | 页面职责 |
|---|---|
| `/` | 首页 |
| `/chat` | AI聊天页面 |
| `/courses` | 课程列表页面 |
| `/life` | 校园生活页面 |
| `/reviews` | 评价列表页面 |
| `/reviews/info/:id?` | 评价详情页面 |
| `/canteen` | 食堂与菜品页面 |
| `/bus` | 校车信息页面 |
| `/userinfo` | 用户资料页面 |
| `/aitest` | AI测试页面 |
| `/admin/users` | 后台用户管理 |
| `/admin/coursecomments` | 后台评论管理 |
| `/admin/courses` | 后台课程管理 |
| `/admin/locations` | 后台地点管理 |
| `/admin/dishs` | 后台菜品管理 |

路由层统一由 `Layout` 组件或后台管理布局承载，实现公共导航、内容插槽和风格统一。

## 3.2 状态管理设计

前端使用 Zustand 管理认证状态。

### 状态内容

- `user`
- `token`
- `isAuthenticated`

### 核心方法

- `setUser()`：设置用户信息和登录态
- `logout()`：清除登录状态

### 持久化设计

认证信息通过持久化中间件保存到本地缓存，保证刷新页面后仍可保留登录态。

## 3.3 接口调用设计

前端通过统一的 HTTP 请求模块封装接口访问能力。

### 设计要点

- 统一 `baseURL`
- 统一设置 `Content-Type`
- 请求拦截器自动附加 `Authorization`
- 响应拦截器统一处理数据和错误

### 接口模块划分

- `userApi.ts`：登录注册
- `userInfo.ts`：用户资料
- `courseApi.ts`：课程数据
- `courseCommentApi.ts`：评论数据
- `canteenLocationApi.ts`：地点数据
- `dishApi.ts`：菜品数据
- `ChatMessagesApi.ts`：聊天会话与消息
- `streamingApi.ts`：流式聊天
- `common.ts` / `fillApi.ts`：上传与智能填充

## 3.4 聊天页面设计

聊天页面是系统复杂度较高的前端模块。

### 页面职责

- 加载历史会话列表
- 创建新会话
- 加载指定会话消息
- 发送用户消息
- 流式展示 AI 回复
- 将流式内容整合为完整聊天记录

### 核心状态

- `conversations`
- `activeConversationId`
- `inputValue`
- `sidebarCollapsed`
- `isNewPage`

### 流式渲染方案

前端通过 `fetch` + `ReadableStream` 读取后端分片输出，逐步拼接 `AiResponse` 并实时更新界面。流式完成后，将完整结果转为正式 `assistant` 消息插入消息数组。

### Markdown 渲染设计

聊天消息使用 Markdown 渲染组件展示，针对未闭合标记场景做了补全处理，以减少流式输出时的闪烁。

## 3.5 后台管理页面设计

后台管理页面采用左右结构：

- 左侧：模块导航
- 右侧：列表、表单、编辑内容区

### 管理模块

- 用户管理
- 评论管理
- 课程管理
- 地点管理
- 菜品管理

各模块设计模式基本一致：

1. 页面初始化拉取列表
2. 用户执行新增/编辑/删除
3. 提交到后端接口
4. 刷新数据并反馈结果

## 4. 后端详细设计

## 4.1 启动设计

后端主启动类 `XiaoYApplication` 使用：

- `@SpringBootApplication`
- `@MapperScan("com.pj.xiaoY.mapper")`

用于自动扫描 Spring 组件和 MyBatis Mapper。

## 4.2 统一返回设计

后端统一使用 `Result` 对象作为响应结构。

### 字段设计

- `success`：是否成功
- `errorMsg`：错误信息
- `data`：响应数据
- `total`：总数或列表条数

### 设计收益

- 统一前后端联调格式
- 降低异常处理复杂度
- 提高接口一致性

## 4.3 Controller 设计

Controller 负责暴露 REST 接口，当前核心模块包括：

- `UserController`
- `UserInfoController`
- `CourseController`
- `CourseCommentController`
- `CanteenLocationController`
- `DishController`
- `ChatMessageByUserController`
- `XiaoYController`
- `UploadController`
- `FillClassController`

### 设计约束

- Controller 不承载过重业务逻辑
- 只做参数接收、调用 Service、封装结果
- URL 命名尽量统一

## 4.4 Service 设计

Service 层负责：

- 业务规则处理
- 数据聚合
- 调用 Mapper、Repository 和工具类
- 处理鉴权上下文数据

### 核心 Service

- 用户服务
- 用户资料服务
- 课程服务
- 评论服务
- 地点服务
- 菜品服务
- 聊天服务
- 上传服务
- AI填充服务

## 4.5 数据访问设计

## 4.5.1 MySQL 访问

结构化业务数据通过 MyBatis-Plus 与 Mapper XML 访问。

### 主要表对象

- `User`
- `UserInfo`
- `Course`
- `CourseComment`
- `Dish`
- `CanteenLocation`
- `CourseSummary`（如后续启用）

### 设计原则

- 实体与表结构字段保持映射一致
- 分页查询统一处理
- 更新操作优先根据 ID 处理

## 4.5.2 MongoDB 访问

MongoDB 用于保存聊天会话与消息历史。

### 主要用途

- 存储会话标题
- 存储用户消息与 AI 回复
- 支撑会话记忆能力

## 4.5.3 Redis 访问

Redis 用于保存用户登录态。

### 设计方式

- `Authorization` 中的 token 作为 key 或映射依据
- value 存储用户信息 JSON
- 拦截器通过 Redis 判断 token 是否有效

## 4.5.4 MinIO 存储

上传文件通过 MinIO 持久化。

### 存储对象

- 用户头像
- 业务附件
- 后续扩展文件资源

## 4.6 鉴权设计

系统通过 `AuthInterceptor` 实现登录校验。

### 工作流程

1. 读取请求头中的 `Authorization`
2. 校验 token 是否存在
3. 从 Redis 查询用户信息
4. 若不存在则返回失败结果
5. 若存在则将用户信息写入线程上下文
6. 请求处理完成后清理上下文

### 当前拦截范围

- 聊天相关接口
- 评论提交接口

## 4.7 AI能力设计

AI能力由 `assistant/` 包中的组件负责封装。

### 当前能力组成

- 普通聊天
- 流式聊天
- 结构化填充
- 嵌入与知识增强扩展能力

### 设计思路

1. 接收用户输入和会话 ID
2. 按需组合提示词、上下文与历史消息
3. 调用模型服务生成结果
4. 普通接口直接返回文本
5. 流式接口以 `Flux<String>` 逐步返回内容

## 4.8 异步与流式支持设计

后端通过 MVC 异步执行器支持异步请求处理。

### 设计目的

- 支持流式接口处理
- 提升长连接请求稳定性
- 避免阻塞主线程

### 线程池策略

- 根据 CPU 核数动态设置线程池大小
- 设置队列容量和超时时间
- 使用调用者运行策略处理拒绝任务

## 6. 数据库逻辑设计

## 6.1 用户相关

### 用户表

保存账号登录信息。

### 用户资料表

保存昵称、联系方式、头像、角色状态等扩展信息。

两者通过 `userinfoId` 或用户主键关系进行关联。

## 6.2 课程相关

### 课程表

保存课程基础信息和统计字段。

### 评论表

保存课程评价内容及用户信息。

## 6.3 食堂相关

### 地点表

保存食堂地点名称等信息。

### 菜品表

保存菜品名称、价格、描述、地点归属等信息。

## 6.4 聊天相关

聊天会话与消息采用 MongoDB 文档结构保存，以便更灵活地存储对话历史。

## 7. 关键设计约束

1. 前后端字段命名尽量统一
2. 统一使用 `Result` 响应格式
3. 受保护接口必须通过拦截器校验
4. 聊天流式能力需优先保证稳定性
5. 文件上传应避免非法路径和空文件问题
