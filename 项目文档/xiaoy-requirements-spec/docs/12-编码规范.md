# 编码规范

## 1. 文档目的

本文档用于统一小Y项目开发过程中的代码风格、命名约定、目录组织、接口规范和提交流程，降低协作成本，提高代码可读性、可维护性和可扩展性。

## 2. 通用规范

## 2.1 基本原则

1. 代码应以“可读性优先”为原则
2. 命名应表达真实业务含义，避免无意义缩写
3. 单个模块职责应尽量单一
4. 不随意修改无关代码或大面积重排格式
5. 公共逻辑优先抽取，避免重复实现
6. 新增功能应优先与现有结构保持一致

## 2.2 注释原则

- 注释用于解释“为什么”，不是简单复述“做了什么”
- 公共类、复杂方法和关键流程应保留必要说明
- 过时注释要及时删除，避免误导
- 不保留大量无意义注释代码

## 2.3 提交原则

- 每次提交应聚焦一个主题
- 提交前确保代码可运行、无明显语法错误
- 提交信息应简洁明确

建议提交信息格式：

- `feat: 新增AI聊天会话列表`
- `fix: 修复评论提交登录校验问题`
- `docs: 更新需求与设计文档`
- `refactor: 重构用户信息接口封装`
- `style: 调整页面样式与布局`

## 3. 前端编码规范

## 3.1 技术栈约定

前端统一基于以下技术栈：

- React
- TypeScript
- Vite
- Zustand
- Axios

## 3.2 目录规范

建议保持如下职责划分：

- `pages/`：页面组件
- `components/`：公共组件
- `services/`：接口调用层
- `store/`：全局状态管理
- `router/`：路由配置
- `types/`：类型定义
- `assets/`：静态资源

### 约束要求

- 页面级逻辑放在 `pages/`
- 跨页面复用组件放在 `components/`
- 所有接口调用统一收敛到 `services/`
- 类型定义优先独立放在 `types/`
- 不在页面中硬编码大量接口路径

## 3.3 命名规范

### 文件命名

- React 组件文件使用 `PascalCase` 或与现有项目保持一致
- 普通工具、服务文件使用 `camelCase` 或语义化英文命名
- 样式文件名需与组件或页面含义对应

示例：

- `ChatPage.tsx`
- `AdminCourses/index.tsx`
- `courseApi.ts`
- `authStore.ts`

### 变量命名

- 普通变量使用 `camelCase`
- 常量使用语义化名称，避免 `a`、`b`、`temp1`
- 布尔值应体现语义，如 `isLoading`、`isAuthenticated`

### 组件命名

- 组件名使用 `PascalCase`
- 名称应与业务含义一致，如 `UserInfoPage`、`AdminUsers`

## 3.4 TypeScript 规范

1. 优先为接口返回值、状态对象、函数参数定义类型
2. 谨慎使用 `any`
3. 公共数据结构统一沉淀到 `types/`
4. 接口响应应定义明确的响应类型
5. 尽量避免把复杂对象类型直接写在组件内部

示例要求：

- 使用 `interface` 或 `type` 定义业务对象
- `Promise` 返回类型应明确
- 状态变量类型应尽量完整

## 3.5 React 组件规范

1. 单个组件职责清晰
2. 页面组件负责页面布局与流程编排
3. 公共组件负责复用能力，不承载过重业务逻辑
4. Hooks 调用顺序必须稳定
5. 副作用逻辑放在 `useEffect` 中处理

### 建议

- 复杂页面按“状态、生命周期、事件、渲染”顺序组织代码
- 长函数应拆分辅助方法
- 可复用逻辑优先抽成自定义 Hook 或工具函数

## 3.6 接口调用规范

1. 禁止在多个页面中重复拼接相同接口路径
2. 所有接口应通过 `services/` 统一封装
3. 认证请求应自动附带 `Authorization`
4. 错误提示应统一处理，避免每个页面重复实现
5. 流式接口需单独封装，不与普通 Axios 请求混用

## 3.7 样式规范

1. 样式文件名与组件/页面保持对应关系
2. 优先使用语义化类名
3. 避免嵌套过深和全局污染
4. 不在 JSX 中写过长内联样式
5. 公共布局和主题风格应保持一致

## 3.8 前端安全规范

1. 不在前端硬编码敏感密钥
2. 不在控制台输出敏感用户信息
3. 上传前应进行基础文件校验
4. 登录态失效时应及时清理本地状态

## 4. 后端编码规范

## 4.1 技术栈约定

后端统一基于以下技术栈：

- Spring Boot 3
- MyBatis-Plus
- MySQL
- Redis
- MongoDB
- MinIO
- LangChain4j

## 4.2 包结构规范

按职责划分包结构：

- `controller`：接口层
- `service`：服务接口层
- `service.impl`：服务实现层
- `mapper`：数据访问层
- `entity`：实体层
- `repository`：Mongo 持久层
- `config`：配置类
- `interceptor`：拦截器
- `common`：通用类
- `utils`：工具类
- `assistant`：AI能力封装

### 约束要求

- Controller 只负责接收和转发，不写复杂业务逻辑
- Service 层负责核心业务处理
- Mapper 只负责数据库访问
- 工具类不依赖具体业务状态

## 4.3 Java 命名规范

### 类命名

- 使用 `PascalCase`
- 类名体现职责，例如：
  - `UserController`
  - `CourseServiceImpl`
  - `AuthInterceptor`

### 方法命名

- 使用 `camelCase`
- 方法名应明确表达行为，如：
  - `queryPage`
  - `uploadAvatar`
  - `getNewMemoryId`

### 变量命名

- 使用 `camelCase`
- 避免拼音与中英混杂
- 避免无意义前后缀

### 常量命名

- 使用全大写加下划线
- 例如：`DEFAULT_PAGE_SIZE`

## 4.4 Controller 规范

1. 路径命名应统一
2. 一个 Controller 只负责一个业务域
3. 使用合适的 HTTP 方法
4. 返回结果统一使用 `Result`
5. 参数校验尽量前置

### 建议

- `GET`：查询
- `POST`：新增或特殊动作
- `PUT`：更新
- `DELETE`：删除

## 4.5 Service 规范

1. Service 接口定义能力边界
2. 复杂逻辑写在 `impl` 中
3. 不在 Service 中直接处理 Web 层细节
4. 多步骤业务应保持事务一致性
5. 业务异常信息应清晰可追踪

## 4.6 Mapper 规范

1. Mapper 方法名应与业务语义一致
2. SQL 与实体字段映射保持一致
3. 复杂 SQL 应加必要说明
4. 避免在 XML 中堆积无用语句

## 4.7 实体与 DTO/VO 规范

1. 数据库实体用于持久化映射
2. 对外响应优先使用 VO
3. 前端输入可按需使用 DTO
4. 不建议将数据库实体直接无差别暴露给前端

当前项目如未完全分离 DTO / VO，可作为后续优化项逐步推进。

## 4.8 异常处理规范

1. 公共异常统一处理
2. 错误信息应简洁明确
3. 不向前端暴露底层堆栈信息
4. 对可预期错误返回可理解提示

## 4.9 日志规范

1. 关键节点应记录日志
2. 不记录明文密码、密钥等敏感信息
3. 错误日志需包含定位问题所需上下文
4. 调试日志在生产环境应可控关闭

## 4.10 安全规范

1. 所有受保护接口必须校验 token
2. 文件上传需校验空文件与非法参数
3. 密码应采用安全存储方案
4. 配置中的密钥应通过环境变量管理
5. Redis、MongoDB、MySQL 的连接信息不得泄露到公开代码或文档

## 5. 数据库与接口规范

## 5.1 数据库规范

1. 表名、字段名应统一风格
2. 主键、外键关系要清晰
3. 时间字段建议统一命名为 `createTime`、`updateTime`
4. 逻辑删除字段统一命名，如 `isDeleted`
5. 枚举含义应在文档中说明

## 5.2 接口规范

1. 接口路径应语义明确
2. 响应格式统一
3. 列表接口支持分页参数
4. 鉴权接口明确认证要求
5. 文件上传接口明确参数说明与限制

## 6. 文档规范

1. 新增模块应同步更新设计文档或需求文档
2. 接口变更应同步更新接口文档
3. 重要配置变更应更新部署文档
4. 对用户可见功能变更应更新用户手册

## 7. 代码评审建议

提交前建议自查以下内容：

1. 是否影响现有功能
2. 是否有明显重复代码
3. 是否补充必要类型或注释
4. 是否有未删除的调试代码
5. 是否有硬编码路径、账号、密钥
6. 是否遵循统一命名和目录规范
7. 是否更新相关文档
